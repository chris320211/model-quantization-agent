# Quant-Execute — venv bootstrap, launch, and runtime fix loop

This skill is the second half of the `model-quantization-agent` pipeline. Its input is a quantization script written by the `quant` skill (or any script that follows the same header convention). Its output is a successfully-quantized model on disk, or — if the failure is non-retryable — a clear explanation of why.

It mirrors the Python package's `executor.py` (background launch + job state under `jobs/<id>/`) and `fix_agent.py` (one-fix-per-attempt repair loop, max 3 attempts).

## Preconditions

- `nvidia-smi` resolves on the host. Bail with a clear message if not (`Bash: command -v nvidia-smi`).
- Python 3.10 is available (`python3.10 --version`). If only `python3` exists and it's >= 3.10, use that.
- The script to run exists at the path the user provides, or at `out/quantize_*.py` if there's exactly one match.

## When to invoke

Trigger on any of:

- "run / execute / launch `out/quantize_*.py`"
- "execute the `<method>` script"
- "kick off the quant"
- "fix and retry job `<job_id>`" (resume a previously-failed run; see Phase 6)

If the user gives no script path and `out/` has multiple `quantize_*.py` files, ask which one. Don't pick silently.

## Inputs

- **Path to script** — e.g. `out/quantize_meta-llama_Llama-2-7b-hf_awq.py`. Filename pattern is `quantize_<safe_model>_<method_id>.py`; `method_id` is the suffix before `.py`.
- **Optional `job_id`** — if the user is resuming a failed job under `jobs/<id>/`, skip Phase 2-3 and go straight to Phase 5.

## Phase 1 — Resolve script and method

1. Verify the script exists. `Bash: test -f <script_path>`.
2. Extract `method_id` from the filename: everything between the last `_` and `.py`.
3. Extract `model_id` from the script header (line starting with `# Model:`).
4. Verify the script parses: `python3 -c "import ast; ast.parse(open('<script_path>').read())"`. If it fails, abort and tell the user to re-run the `quant` skill or fix the script manually.

## Phase 2 — Bootstrap the method venv (skip if already provisioned)

Target layout (matches the Python agent's `executor.py` so the `quant-agent jobs ...` CLI keeps working):

```
.venvs/<method_id>/
├── bin/python              # method's interpreter
├── repo/                   # method's GitHub repo (cloned by `quant`)
└── .installed              # marker, written when bootstrap succeeds
```

Checks:

- If `.venvs/<method_id>/.installed` exists: skip bootstrap.
- Otherwise: parse the install commands from the script header (between `# Before running, install the method into a clean venv:` and `# Then: python ...`). The header is written by the `quant` skill in this exact format.

Bootstrap sequence (idempotent):

```bash
python3.10 -m venv .venvs/<method_id>
source .venvs/<method_id>/bin/activate
pip install --upgrade pip wheel
pip install --index-url https://download.pytorch.org/whl/cu121 torch==2.3.1
pip install transformers accelerate safetensors sentencepiece
# then each method-specific line from the script header, e.g.:
pip install autoawq
deactivate
touch .venvs/<method_id>/.installed
```

Run via `Bash` with a generous timeout (`timeout: 900000` = 15 min — torch wheels are big). If a step fails:

- **Network / PyPI flake** → retry the failing step once.
- **Compile failure on a method-specific package** → surface the error, stop. This is a setup problem, not a runtime fix.
- **`TORCH_INDEX` override needed** (CUDA != 12.1) → tell the user to set `TORCH_INDEX=https://download.pytorch.org/whl/cu118` (or matching) and re-invoke.

If the script header lacks the install block, ask the user for the install commands explicitly. **Do not invent them.**

If the cloned repo at `.venvs/<method_id>/repo/` is missing but the script references it (Style B subprocess wrapper), clone it now: `git clone --depth 1 <repo_url> .venvs/<method_id>/repo`. Get `repo_url` from `reference/methods.yaml` if available, or by looking up the method id in `../quant/reference/methods.yaml` from the sibling skill, or by asking the user.

## Phase 3 — Launch the job in the background

Mirror `executor.launch` in `src/quant_agent/executor.py`:

1. Generate a `job_id`: `Bash: date -u +%Y%m%dT%H%M%SZ-$(openssl rand -hex 3)`.
2. Create `jobs/<job_id>/` and snapshot the script: `cp <script_path> jobs/<job_id>/script.py`.
3. Launch with `setsid` so it survives an SSH disconnect (this is the same trick `executor.py` uses):

```bash
setsid bash -c '
  .venvs/<method_id>/bin/python jobs/<job_id>/script.py \
    > jobs/<job_id>/stdout.log 2> jobs/<job_id>/stderr.log
  echo $? > jobs/<job_id>/exit_code
' &
echo $! > jobs/<job_id>/pid
```

Run this with `Bash run_in_background: true` so Claude Code returns immediately. Capture the PID.

4. Write `jobs/<job_id>/meta.json` with the same shape as `executor.JobMeta`:

```json
{
  "job_id": "<id>",
  "method_id": "<method>",
  "model_id": "<model_id>",
  "venv": "<method>",
  "script_path": "jobs/<job_id>/script.py",
  "output_dir": "out/",
  "pid": <pid>,
  "started_at": "<ISO-8601 UTC>",
  "finished_at": null,
  "exit_code": null,
  "status": "running",
  "parent_job_id": null,
  "attempt": 1
}
```

(On a fix-retry attempt, set `parent_job_id` to the previous failed `job_id` and bump `attempt`.)

5. Print the `job_id` and the path to the log files so the user can `tail -f jobs/<id>/stdout.log` from another shell if they want.

## Phase 4 — Wait for completion

Poll `jobs/<job_id>/exit_code` every 30s using `Monitor` on the background Bash launch, or a simple loop:

```bash
until [ -f jobs/<job_id>/exit_code ] || ! kill -0 <pid> 2>/dev/null; do sleep 30; done
```

While waiting, every 30s `Read` the last 40 lines of `jobs/<job_id>/stdout.log` and `stderr.log` to surface progress to the user. **Do not poll faster than 30s** — the cache TTL math from `ScheduleWakeup` applies here too, and quant jobs print sparingly.

Outcomes:

- `exit_code` file contains `0` → success. Update `meta.json` (`status: "completed"`, `finished_at`, `exit_code: 0`). Go to Phase 7.
- `exit_code` contains non-zero → runtime failure. Update `meta.json` (`status: "failed"`). Go to Phase 5.
- PID is gone but no `exit_code` file → killed externally. Update `meta.json` (`status: "killed"`). Tell the user, stop.

## Phase 5 — Fix loop (max 3 attempts)

This phase is a verbatim port of the `fix_agent.py` playbook (`src/quant_agent/fix_agent.py:36-79`). You are now the Fix agent for `job_id <id>` (`<method_name>` / `<model_id>`), attempt `<attempt>` of 3.

You must NOT switch methods — the user already chose `<method_name>`. You apply **one** fix per attempt and relaunch.

### Step 5a — Read logs

`Read jobs/<job_id>/stdout.log` and `Read jobs/<job_id>/stderr.log` (last 200 lines each). Identify the failing step and the root-cause error line.

### Step 5b — If you plan to edit the script, read it first

Run `Read jobs/<job_id>/script.py` before any `Edit` call so you know the exact text. `Edit`'s `old_string` must match the file content exactly once — guessing from the traceback is unreliable.

### Step 5c — Classify and pick ONE fix

| Symptom in stderr | Fix |
|---|---|
| `ImportError` / `ModuleNotFoundError: No module named '<pkg>'` | `Bash: .venvs/<method_id>/bin/pip install <pkg>` (or `==<version>` if the traceback references a version) |
| `requires X but found Y` (version conflict) | `Bash: .venvs/<method_id>/bin/pip install <pkg>==<X>` |
| `No module named '<method>_inference_engine'` or other CUDA-extension miss | `Bash: cd .venvs/<method_id>/repo/<subdir> && TORCH_CUDA_ARCH_LIST=<compute_capability> ../../bin/python setup.py install` — set `TORCH_CUDA_ARCH_LIST` to the GPU's compute capability (look it up from `../quant/reference/gpu_specs.yaml` if needed) |
| `argparse: unrecognized arguments` / wrong flag | `Edit jobs/<job_id>/script.py` — replace the offending flag string with the real one (read the cloned example file at `.venvs/<method_id>/repo/...` to find it) |
| HF `401`/`403` on a gated model | **Non-retryable.** Tell the user to set `HF_TOKEN` and re-invoke. **Do not relaunch.** |
| `OutOfMemoryError` / `CUDA out of memory` | **Non-retryable** at the same bit width. Tell the user to (a) pick a smaller model, (b) increase bits in the script (lower-bit usually means *less* memory, so check whether a higher-bit option in `methods.yaml` actually fits worse — usually not), or (c) move to a larger instance. **Do not relaunch.** |
| Missing `HF_TOKEN` (`401` with `gated_repo`) | **Non-retryable.** Tell the user to `export HF_TOKEN=...` and re-invoke. |
| `No space left on device` | **Non-retryable.** Tell the user to free EBS or remount with more space. |
| Anything else not above | Inspect the cloned repo (`Read .venvs/<method_id>/repo/<file>`) or the upstream README (`WebFetch`) before guessing. If you still can't classify, mark non-retryable and stop. |

**Apply exactly one fix.** Prefer venv surgery (pip install / setup.py install) over script edits. If unsure, inspect the cloned repo before editing.

### Step 5d — Relaunch (only if you applied a fix)

If the fix was non-retryable: stop, write a one-sentence explanation, set `meta.json` of the failed job to `status: "failed"` (already done in Phase 4), don't launch a new job.

If the fix was applied: relaunch by repeating Phase 3 with:

- A new `job_id`.
- `parent_job_id` set to the previous failed `job_id`.
- `attempt` incremented (the previous job's attempt + 1).

Then return to Phase 4. Repeat until success or `attempt > 3` or non-retryable.

When `attempt > 3` without success: stop. Tell the user the fix loop exhausted its budget and link to all attempts (`jobs/<id1>`, `jobs/<id2>`, `jobs/<id3>`).

## Phase 6 — Resuming an existing failed job

If the user invoked the skill with a `job_id` (e.g. "fix and retry `<id>`"):

1. `Read jobs/<id>/meta.json`. Confirm `status` is `failed`. If `running` or `completed`, tell the user nothing to do.
2. Pull `method_id`, `model_id`, `script_path`, `attempt` from meta.
3. Skip Phase 2 (venv exists if the job ran at all).
4. Jump straight into Phase 5 with `attempt` = (meta.attempt + 1).

## Phase 7 — Success handoff

When `exit_code` is `0`:

1. Print the quantized output directory. The script writes to `./quantized/<method_id>-<safe_model>/`. Confirm with `Bash: ls -lah quantized/<method_id>-*/`.
2. Print the disk size of the quantized model (the script's last printed line — grep stdout.log for it, or `du -sh quantized/<method_id>-*/`).
3. Print the chain of `job_id`s if there were retries (parent → child → ...).

That's the end of the skill. No upload, no inference test — those are out of scope.

## Guardrails

- **Don't switch methods.** The user already picked. If the chosen method genuinely can't run on the hardware (OOM, missing compute capability), say so and stop — don't silently swap to another method.
- **One fix per attempt.** Resist the urge to combine an `install_method_venv` and a script `Edit` in the same attempt; you lose the ability to know which one mattered if the next run still fails.
- **Don't tail logs faster than every 30s.** Quant jobs are I/O-light — fast polling burns the prompt cache without surfacing more information.
- **Don't auto-retry non-retryable failures.** Gated-model 401s, OOMs, and disk-full are user-environment problems; relaunching them just consumes budget.
- **Don't run on a non-CUDA host.** The skill's preconditions block this; don't try to fall back to CPU.
- **Keep `jobs/<id>/` intact.** It's the contract with `quant-agent jobs list/status/logs/kill` from the Python CLI, so users can poke at jobs from a separate shell.

## Cross-skill contract with `quant`

Inputs this skill expects from `quant`:

- A script at `out/quantize_<safe_model>_<method_id>.py`.
- A header block of the form documented in `../quant/SKILL.md` (Phase 4 step 6) that lists the install commands.
- A `Method:` line, a `Model:` line, and a `Target:` line in the header.

If any of those are absent (e.g. the user wrote the script by hand), ask for the missing pieces rather than guessing.
